What OAuth really is?

OAuth is an authentication protocol that allows you to approve one application interacting with one another on your behalf without giving out your password.

How OAuth works -

How OAuth works is by assigning user authentication to the service that hosts the user account, and authorizing 3rd-party applications to access the user account.

History of OAuth (1.0 versus 2.0).
OAuth1.0 starts around the year 2008 it was used by this popular socia media website called Twitter because of that, Twitter forced all 3rd-party apps to use their 0Auth 1.0 implementation then few years later in October 2012 that is when 2.0 came out. It was said that 2.0 is much simplier to implement than 1.0 with its crypto-underpinnings.
Why OAuth is popular?
Because it is more secure than sharing credentials to strangers and allows users to utilize services across multiple platforms. OAuth also offers a good balance with security for your webpage.
End to end data flow of an OAuth API.

OAuth defines four roles and they are Resource owner, Resource server, a client, and the authorization server. The four defined roles makes uses of the end to end data flow of an OAuth API.

Authorization code flow

The client needs to perform two steps to obtain the token, the first involving the browser, the second a back-channel request. That's why this series of steps is also called the code flow.

Step 1 - Code Flow

The client initiates the code flow by redirecting the browser to the authorization server with an authorization request.

Step 2 - Code Flow

The authorization code is an intermediate credential, which encodes the authorization obtained at step 1. It is therefore opaque to the client and only has meaning to the authorization server. To retrieve the access token the client must submit the code to the authorization server, but this time with a direct back-channel request.
